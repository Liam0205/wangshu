# Guide:多文档并行起草工作流

> 适用:一次任务要新增/重写多篇相互引用的设计文档时。
> 来源:`memory/reflections/2026-06-11-design-doc-completion.md`(19 篇 P1+P2-P5 文档集实战验证);收尾与提问两节来自 `memory/reflections/2026-06-11-design-review-round.md`(7 项决策评审轮实战)。

## 核心协议(三条)

1. **并行起草、单点收口**。每篇由独立子代理起草,prompt 必须含:必读上游文档清单 + 风格基线 + 回填请求约定。主助理逐篇核验,**跨文档回填只由主助理统一执行**——分散给子代理会让「A 请求 B 增字段」悬空。
2. **回填请求节协议**。下游文档需要上游增补字段/布局时,**不假设已存在**,在自己文档内写「对 XX 的回填请求」节;主助理统一兑现后改标「已兑现」。请求与兑现状态都留在文档内,可审计、不丢失。
3. **指定唯一验收口径收口点**。所有文档把「待定口径」标「待 XX 定」,指向预先指定的一篇收口文档(P1 的收口点是 `docs/design/p1-interpreter/12-testing-difftest.md` §10 总表),由它逐条裁决。**分散提出、集中收口**,避免口径在多篇间打架。

## 子代理失败恢复纪律

- 子代理中断后,续接前**先核验文件是否已落盘**(`wc -l` + 看头尾);已落盘则续写,勿重写。
- 同一篇连续失败 2-3 次,改由**主助理亲自撰写**——单次大块 Write 比长会话子代理抗中断。

## 规范类内容纪律

- 基于已有公开规范知识直接撰写,不抓取第三方源码;
- 不确定的精确措辞(如错误信息)一律标「待差分核对」,**不编造**。

## 收尾:主动盘点不确定决策

大型起草任务收尾时,主助理按「**影响 × 不确定度**」筛出设计期自行定下来但没把握的决策,**分层呈现**(影响开工形式 / 依赖外部数据 / 低风险已记录),交用户逐项裁决。首次实战 7 项命中 4 项需改(含 1 项方向性反转),不盘点就是实现期返工。裁决完成时**立即 grep 全部受影响文档同步引用点**——单点改动多点同步是完成的主要工作量,逐项即时同步避免收尾遗漏。

## 向用户提问必须自包含(AskUserQuestion 契约)

提问 UI 里用户**看不到之前消息的长正文**。问题描述必须独立装下全部必要背景:矛盾是什么 + 每个选项的具体后果,**绝不写「见上文」或留指代**。这是工具交互契约而非表达风格;发问前自检:脱离上文能否独立理解。违反的实际代价:同一问题被回「看不懂/看不到正文」三次。
