# 设计文档评审决策轮(7 项不确定决策逐项裁决)

- **日期**:2026-06-11
- **任务类型**:设计完成后的主动决策盘点 + 用户逐项裁决 + 修订落盘

## 任务

设计文档集齐备后,主助理主动盘点「设计期拍板但不确定」的 7 项决策,逐项与用户讨论并落地:4 项修订(验收 oracle 改官方 5.1.5+gopher 豁免、luac 同构降软承诺、ColInt64 超界报错、stdlib 反转为对齐 gopher-lua 提供面+三层禁用)、3 项维持现状(协程路线 B、pairs 混合口径、P3 协程不升层)。用户中途追加一条产品需求(方便禁用标准库),当场设计进 10 §12.1。

## 预期 vs 实际

- 预期:确认性质的过会,多数维持原状。
- 实际:7 项中 1 项方向性反转、3 项实质修订——这些若不主动盘点,会变成实现期返工。另收获 1 条计划外产品需求。

## 做对了什么(可复用模式)

1. **「完成后主动盘点不确定决策」值得固化为流程**。大型设计任务收尾时,主助理按「影响 × 不确定度」筛出设计期自行拍板但没把握的决策,分层呈现(影响 P1 开工形态 / 依赖宿主数据 / 低风险已记录),用户只需逐个裁决。本次 7 项命中 4 项需改,证明收益远超成本。
2. **裁决后立即落盘并 grep 同步所有引用点**。每项裁决落地时不只改主文档,立刻 grep 全部受影响文档(roadmap 措辞改动同步 architecture/00-overview/llmdoc 四处;ColInt64 同步 §3.2 表格、§4 伪码、12 口径表新增第 26 条)。**单点改动多点同步是评审落地的主要工作量**,逐项即时同步避免收尾时遗漏。
3. **识别用户插话中的隐性新需求**。「应方便禁用标准库」是在修订 os 默认面时插进来的,不是对当前修订的评论而是新产品需求——识别出来当场设计(LibsSafe 预设 / Libs 位掩码 / Exclude 函数级三层),避免丢失在对话流里。

## 什么出了问题 / 根因

- **AskUserQuestion 不自包含,stdlib 一问三次被回「看不懂/看不到正文」**。根因:提问 UI 里用户看不到之前消息的长正文,而该问题的背景(矛盾是什么、各选项后果)全写在上一条消息里,问题本体只剩指代。教训:**提问描述必须自包含装下全部必要背景**——矛盾陈述 + 每个选项的具体后果,绝不写「见上文」。这是工具交互契约,不是表达风格问题。
